## Summary

Build a single documentation area at `/docs/` serving all authenticated users with role-based section visibility. Regular users see 7 usage sections (SSR, publicly accessible). Admin users additionally see 3 admin-only sections rendered client-side with admin role check — non-admins attempting direct URL access are redirected to `/docs/`. No new backend endpoints, no new npm packages, no new infrastructure. Estimated ~5 days, 1 sprint.

## Key Technical Decision: Admin Role Check

The `(admin)` route group layout performs the admin check client-side:

```
+layout.svelte (admin group):
  1. Import user store from lib/stores/auth.ts
  2. $effect: when auth resolves (user !== undefined):
     - If user is null → redirect to /login/
     - If user.is_admin is false → redirect to /docs/
     - Else → render <slot /> (admin content)
  3. Show loading state while auth is resolving
```

The `is_admin` flag is sourced from the existing Firebase user profile or custom claims. The specific check mechanism (custom claim vs Firestore profile fetch) is confirmed during implementation — the pattern matches the existing `require_admin` backend dependency.

The `(admin)` group `+layout.ts` sets `export const ssr = false` so SvelteKit never server-renders admin pages — no risk of content leaking during SSR.

## Content Outline Per Section

### `/docs/` — Home
- Intro: what The Loop is (2 sentences)
- `PersonaPicker`: 6 role cards. Selecting a role highlights relevant section cards below with `ring-accent`.
- Feature index: grid of section cards. Regular users see 7 cards; admin users see 10 (7 + 3 admin-only, visually grouped with a subtle "Administration" label).

### `/docs/getting-started/`
Prerequisites → Create first incident → Navigate dashboard → Invite team

### `/docs/incidents/`
Lifecycle (create→open→investigating→resolved→closed) → Fields explained (title, severity, category, team, status) → Responders → Action Items → Timeline events → Attachments → Closing checklist

### `/docs/postmortems/`
When to write → Template catalog (5-Whys, Fishbone, Timeline, Pre/Mortem, OODA, PDCA) → Fields → Locking (irreversible warning) → AI summary

### `/docs/analytics/`
Dashboard overview → 8 KPI cards explained → Filters (period/team/category/severity/status) → Drill-down → PatternTimeline → SeverityTrend chart → Rule Effectiveness table

### `/docs/semgrep/`
Architecture → Full GitHub Actions workflow YAML (CodeBlock) → Version pinning → Fallback `.bak` behavior → ERROR vs WARNING → Scan history → Local testing commands

### `/docs/api-keys/`
What it is (scope, `tlp_` prefix) → Creating via UI → Using in CI (`Authorization: Bearer tlp_…`) → Project whitelist → Rotating / revoking

### `/docs/rules/`
Version history (v0.1.0→v0.4.0, rule counts) → Rule browser `/rules/latest/` → Severity model → 10 languages covered → Rule ID format → Requesting a new rule

### `/docs/administration/` *(admin only)*
Team management → Rate limits (60 req/min analytics, 5 req/min waitlist) → API base URL → Single-environment model → Firebase project `theloopoute`

### `/docs/security/` *(admin only)*
Auth tiers table (Anonymous/API key/Firebase JWT/Admin) → Token formats → Security headers (HSTS, CSP, X-Frame-Options) → Firebase Security Rules → Secrets management (GCP Secret Manager) → Cloud Audit Logs

### `/docs/api-reference/` *(admin only)*
Base URL + auth header → Endpoint index table (Method / Path / Auth Tier / Description) covering all groups (Health, Incidents, Sub-resources, Postmortems, Analytics×6, Rules×3, API Keys×4, Scans×2) → List envelope `{"items":[],"total":N}` → Error format `{"detail":"..."}` → Rate limit headers

```svelte

  <title>{pageTitle} — The Loop Docs</title>
  <meta name="description" content="..." />
</svelte:head>
```

All routes respect `trailingSlash: 'always'` from global `+layout.ts`. No overrides.
